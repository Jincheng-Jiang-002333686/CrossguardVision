## 2. Environment gotchas (cost real time — see [[pi-deployment]])

1. **`pip install ultralytics` pulls the WRONG torch on a Pi.** The default aarch64 PyPI wheel
   for torch ≥2.4 is the **server-ARM / CUDA "sbsa" build** (dragged in ~2.5 GB of `nvidia-*`
   libs) and **crashes with `Illegal instruction` (SIGILL)** — the `ldaddal` atomic in
   `libarm_compute*.so` isn't supported by the Pi's Cortex-A72. The geometry `--selftest`
   (cv2+numpy) passed while `model.predict` SIGILL'd → isolated it to torch.
   **Fix:** pin the last good combo —
   `pip install "numpy<2" "torch==2.3.1" "torchvision==0.18.1" "opencv-python<4.10"`
   (`numpy<2` + `opencv<4.10` because torch 2.3.1 is built on the NumPy-1 ABI).
   Ref: pytorch/pytorch#132032, #97226, #109312 (2.3.1 works, 2.4.0+ regressed).
2. **picamera2 isn't pip-installable** (libcamera C bindings are apt-only). Exposed it to the
   isolated venv by setting `include-system-site-packages = true` in `venv/pyvenv.cfg`
   (one-line, non-destructive; venv's own torch/numpy still win).
3. Inference must use **`--device cpu`** (spatial_reasoning's default is `"0"` = CUDA).

## 3. Pi latency — the decisive deployment metric (FINALLY measured)

Steady-state, live from the camera, imgsz 640, batch 1, CPU. Each frame = full pipeline
(model forward + mask assembly + spatial geometry + annotate + JPEG encode).

| Runtime | ms/frame | FPS | Notes |
|---|---:|---:|---|
| torch 2.3.1 (CPU) | ~1450 | 0.7 | baseline |
| **NCNN (fp16=0, optlevel 2)** | **~515** | **1.9** | **2.8× faster, same model/accuracy** |

- NCNN export ran **on the Pi** (`yolo export ... format=ncnn imgsz=640`; pnnx auto-installed
  fine on aarch64). Output `yolo11n-seg-crossguard_ncnn_model/` (11.0 MB). Loaded directly by
  `live_camera.py` via `--weights <dir>` — no code change.
- This is a **Pi 4 / A72** number. A Pi 5 (A76) would be ~2-3× faster again.
- ⚠️ **Not directly comparable to the 2026-06-23 §13 CPU table** (that was imgsz **896** on a
  4-thread x86 laptop = 145.6 ms / 6.9 FPS for 11n). This run is imgsz **640** on real Pi-4 ARM
  via NCNN. Different size + hardware + runtime → it's a new, standalone data point.

## 4. Functional validation

Live feed detects **1–4 persons/frame** reliably. All classified `low_risk` — **correct**:
the indoor test scene has **no crosswalk**, so the distance map is None and every person
defaults to `low_risk` by design. Risk-level classification (high/medium) still to be
visually confirmed **with a crosswalk in frame** (point at a real crosswalk or a printed one).
