### Two performance gotchas (cost real time — see [[training-env-gotchas]])
1. **First launch was data-pipeline bound** (~340 s/epoch). Cause: **169 / 376 train
   images are 4K (3840×2160)** and the run was launched without `--cache`, so every
   epoch re-decoded ~the whole set on CPU. **Fix: `--cache`** (RAM) — ultralytics caches
   images downscaled to the 896 long-side (only **0.5 GB**), dropping epochs to **~48 s**.
2. **Thermal throttling** (laptop 3070). After ~12 min sustained load the GPU heat-soaks
   to ~88 °C and the driver cuts the SM clock **2100 → 210 MHz** (`sw_thermal_slowdown`
   Active) → epochs slowed back to ~340 s. Not fixable in software (non-admin; laptop
   power limit locked). **Improving cooling** (max-fan profile + elevate) brought it to
   an oscillating ~48–100 s/epoch (avg ~65 s) and the full run finished in 2.51 h.

2. **The `m` scale is not worth it.** `26m` has ~2.4× the params / ~3.6× the FLOPs of `26s`

4. **Safety nuance:** `11s` has the best crosswalk **recall** (0.910 vs `26s` 0.864) — it

## 9. Training hyperparameters (verified identical across all 4 runs)

We never set an LR; `optimizer=auto` **overrode** the nominal `lr0=0.01` and chose:
- **AdamW, `lr0=0.001667`, momentum 0.9** (identical for 26n/26s/11s/26m — confirmed in every
  `*_train.log`). ⚠️ `args.yaml` shows `lr0: 0.01` but that is NOT what ran.
- Schedule: `cos_lr=False` → **linear** decay to `lr0×lrf` = 0.001667 × 0.01 = **1.67e‑5**;
  **3-epoch warmup**; weight_decay 5e‑4.
- **`nbs=64` → gradient accumulation makes the *effective optimizer batch = 64* regardless of
  `--batch`** (26m batch2→accum32, s-models batch4→accum16, 26n batch8→accum8). So `--batch`
  only changes BatchNorm micro-batch + memory, NOT the optimizer math.

## 10. Verified deep-dive corrections (adversarially checked vs the run files)

- **GPU "inf" ms is forward-pass only.** `11s` vs `26s` are **tied within noise** on GPU
  (n=1, throttling; `11s` measured 9.3 / 12.9 / 17.1 ms across runs — swing ≫ the 1.5 ms gap;
  one run even shows `26s` faster). My earlier "batch‑1 / layer‑depth" mechanism was wrong
  (measurement was batch‑8, compute‑bound; fused‑FLOPs gap is only ~4%: 34.1 vs 32.8).
- **NMS-free *does* help `26s`** — post-proc 0.6 vs `11s` 1.2 ms (clean run; 1.9 vs 7.4 ms in
  the training-val). End-to-end this can make `26s` faster; it's **decisive on CPU/Pi**, not GPU.
- **`26m`'s batch‑2 is NOT a real confound** (effective batch 64 via grad-accum; identical
  AdamW). `26m` ≈ `26s` overall (mask mAP50-95 0.511 vs 0.510); the 0.024 crosswalk-mask gap is
  **plausibly within noise** (single seed, 317 correlated crosswalk instances). Why bigger ≠
  better here: **proto-mask resolution ceiling (`mask_ratio=4`)** caps crosswalk-mask sharpness
  regardless of backbone size, + tiny-data **saturation** (26m just overfits faster).
- **Whole comparison is n=1 seed** → no sub‑0.03 mAP gap is statistically established.

**Findings (these REVISE §7/§10/§12):**
1. **YOLO26's NMS-free advantage does NOT pay off for this SEG task.** Postproc saving is only
   **~1–2 ms** (n 0.75 / s 1.53 / m 1.96). Reason: seg postproc (~18–24 ms on CPU) is dominated by
   **mask proto assembly + upsample**, identical for both architectures; NMS is only ~1–2 ms of it.
   ⚠️ This **refutes** the §10 "NMS-free is decisive on CPU/Pi" note — that was reasoned from
   *detection* behavior + GPU postproc deltas; the actual seg-CPU measurement contradicts it.
2. **Forward pass dominates CPU latency (~85–96%)** → CPU speed tracks **FLOPs**. YOLO11 has
   equal-or-fewer FLOPs at every scale → **YOLO11 is fastest at every scale on CPU**. The s-pair even
   flips *toward* 11s (282 vs 304 ms), opposite of the GPU reading — the ~24 ms forward deficit
   swamps 26s's ~1.5 ms NMS saving.
3. **CPU ≈ 25–33× slower than this GPU**; a real Pi (weaker ARM) is several× slower again → only
   **n-scale** is plausibly near-interactive on a Pi; **m-scale is off the table** for real-time (~0.7 s
   here → likely >2 s/frame on a Pi).

**REVISED recommendation (supersedes §12's "11n vs 26s, decide on latency"):**
**`yolo11n-seg` is the deployment pick** — fastest on **both** CPU and GPU, smallest, AND best
crosswalk-mask mAP50 point estimate; it wins or ties on every Pi-relevant axis. `26s`'s remaining
case is purely **mask tightness** (best mAP50-95 0.637 + precision) at ~2× CPU latency — choose it
only if strict-IoU mask quality outranks speed. Recall caveat unchanged: `11n` crosswalk recall
0.886 is lowest of the 11 family → if "never miss a crosswalk" dominates, use `11s` (recall 0.910,
~2× latency) or lower `11n`'s conf threshold. If more speed is needed, drop **imgsz** (a separate
